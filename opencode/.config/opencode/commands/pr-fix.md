---
description: Fix PR review comments, resolve threads, and re-request review
---
Address the review comments on the current branch's pull request, then resolve all comment threads and re-request review.

If the user provided a PR number as an argument, use that: `$ARGUMENTS`. Otherwise detect it from the current branch.

## Steps

### 1. Detect the PR

```sh
gh pr view --json number,url,headRefName,baseRefName,reviewRequests
```

If `$ARGUMENTS` is a number, use `gh pr view $ARGUMENTS ...` instead.

### 2. Fetch review comments

Fetch all review comment threads including their thread IDs and resolution status:

```sh
gh api graphql -f query='
  query($owner: String!, $repo: String!, $pr: Int!) {
    repository(owner: $owner, name: $repo) {
      pullRequest(number: $pr) {
        reviewThreads(first: 100) {
          nodes {
            id
            isResolved
            comments(first: 10) {
              nodes {
                id
                databaseId
                author { login }
                body
                path
                line
                originalLine
              }
            }
          }
        }
        reviews(first: 20) {
          nodes {
            author { login }
            state
          }
        }
        latestOpinionatedReviews(first: 10) {
          nodes {
            author { login }
            state
          }
        }
      }
    }
  }
'
```

Fill in `$owner`, `$repo`, and `$pr` from the repo context (use `gh repo view --json owner,name`).

### 3. Understand the current changes

Read the PR diff to understand the full context:

```sh
gh pr diff
```

### 4. Implement the requested changes

For each **unresolved** review thread:
1. Read the comment body and the file/line it references.
2. Open the referenced file and understand the context.
3. Make the requested change. Use your judgment — if a comment is a question or a "nit" that doesn't require a code change, skip it (you'll still reply and resolve it).

### 5. Commit and push

After all changes are made:

```sh
git add -A
git commit -m "Address PR review feedback"
git push
```

If there are no changes to commit (all comments were non-actionable), skip this step but still proceed to resolve and reply.

### 6. Reply to and resolve each comment thread

For each **unresolved** thread from step 2:

**Reply** to the thread with a short message. Use the `in_reply_to` parameter with the **last comment's `databaseId`** in that thread:

```sh
gh api --method POST repos/{owner}/{repo}/pulls/{pr}/comments \
  -f body="Addressed in latest push." \
  -F in_reply_to=<last_comment_database_id>
```

If the comment was non-actionable (a question, acknowledgment, or nit you chose not to change), reply with something appropriate like "Acknowledged, thanks!" or "Good catch, though I think the current approach is fine because..." — keep it brief.

**Resolve** the thread via GraphQL:

```sh
gh api graphql -f query='
  mutation($threadId: ID!) {
    resolveReviewThread(input: { threadId: $threadId }) {
      thread { id isResolved }
    }
  }
' -f threadId="<thread_id>"
```

Do this for every unresolved thread.

### 7. Re-request review

From the `latestOpinionatedReviews` data in step 2, collect the usernames of reviewers whose latest review state is `CHANGES_REQUESTED`. Then re-request review from each:

```sh
gh api --method POST repos/{owner}/{repo}/pulls/{pr}/requested_reviewers \
  -f 'reviewers[]=<username>'
```

If there are multiple reviewers, include them all in one call:

```sh
gh api --method POST repos/{owner}/{repo}/pulls/{pr}/requested_reviewers \
  --input - <<< '{"reviewers": ["user1", "user2"]}'
```

### 8. Report

Summarize what was done:
- How many threads were resolved
- What code changes were made (if any)
- Who was re-requested for review
- The PR URL for easy access

## Rules

- Do NOT force push.
- Do NOT modify files unrelated to the review comments.
- Do NOT skip resolving threads — resolve every unresolved thread, even if no code change was needed.
- If a review comment is unclear or contradictory, use your best judgment and note it in your reply.
- Keep commit messages concise.
- Do not add any "Generated by" or "Co-Authored-By" trailers.
